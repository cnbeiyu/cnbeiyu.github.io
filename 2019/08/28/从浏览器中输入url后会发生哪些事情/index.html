从浏览器中输入URL后会发生哪些事情

Posted on 2019-08-28 |

URL处理

当用户在地址栏中输入一个url后，浏览器首先会对输入内容进行判断，如果不符合url格式，则发送搜索请求，如 https://www.google.com/search?q=%s ，若符合URL格式则对URL加上协议，变为完整的URL，如 https://www.cntwz.com。

DNS解析

浏览器获取到一个合格的URL后，需要知道URL对应服务器的IP地址才能发送请求，浏览器首先会向本地内存Cache中发起查询，如果没有则会向本地hosts文件发起查询，若还是没有，则会向自己的DNS服务器发起查询，然后dns服务器会以此通过根域名服务器、顶级域名服务器、权威域名服务器进行递归查询。在查阅相关资料后发现，DNS的内部逻辑远比我想象的复杂的多，之后会另开一篇文章单独写DNS原理（埋坑）。

在发起网络请求前，浏览器先会对浏览器缓存中查询是否有需要请求文件的缓存标识和结果，浏览器每次拿到返回的请求结果都会将该结果和缓存标识存入浏览器缓存中，如Chrome的缓存存放于.User\AppData\Local\Google\Chrome\User Data\Default\Cache，当缓存中有需要请求的文件时，浏览器会拦截请求，直接返回该资源的副本并且结束请求。浏览器缓存可以提升访问效率，是实现快速资源加载的重要组成部分，如果没有查找到缓存，就会进入接下来的网络请求过程了。

建立TCP连接

因为HTTP协议是应用层协议，先需要TCP协议将他发送到网络上，所以在 HTTP 工作开始之前，浏览器需要通过 TCP 与服务器建立连接。所以接下来又到了熟悉的TCP三次握手环节，发送端首先发送一个带SYN（synchronize）标志的数据包给接收方，接收方收到后，回传一个带有SYN/ACK(acknowledegment)标志的数据包以示传达确认信息。最后发送方再回传一个带ACK标志的数据包，代表握手结束。在这过程中若出现问题中断，TCP会再次发送相同的数据包。
TCP可以简单理解为：
客户端：“你好，有你的快递，请问在家吗？”
服务器：“在的，可以送来。”
客户端：“好的。”

断开TCP连接

在完成一次HTTP请求后，服务器并不是马上断开与客户端的连接。在 HTTP/1.1 中，Connection: keep-alive 是默认启用的，表示持久连接，以便处理不久后到来的新请求，无需重新建立连接而增加慢启动开销。在Nginx 中，持久连接超时时间默认值为75秒，如果75秒内没有新到达的请求，则断开与客户端的连接。同时，浏览器每隔 45 秒会向服务器发送 TCP keep-alive 探测包，来判断 TCP 连接状况，如果没有收到 ACK 应答，则主动断开与服务器的连接，如果收到ACK应答，则就到了熟悉的四次挥手环节，双方关闭连接。HTTP keep-alive和TCP keep-alive虽然都是一种保活机制，但是它们完全不相同，一个作用于应用层，一个作用于传输层。
